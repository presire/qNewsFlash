# qNewsFlash 非機能要件

## 1. 性能要件

### 1.1 処理性能

- **NFR-001**: ニュース記事の取得処理は非同期で実行され、他の処理をブロックしないこと
  - QNetworkAccessManagerによる非同期HTTP通信
  - シグナル・スロット機構による非同期処理
  - UIスレッドをブロックしない設計
  - レスポンス待機中も他の処理を継続可能

- **NFR-002**: 複数のニュースソースからの記事取得を並行して処理できること
  - 各ニュースソースを独立した非同期リクエストで処理
  - 同時実行可能なHTTPリクエスト数: 最大10件
  - リクエストキューイング機能
  - タイムアウトエラーが他のソースに影響しない

### 1.2 タイムアウト設定

- **NFR-003**: 設定ファイルやログファイルへのアクセスは、適切なタイムアウト（5秒）を設定すること
  - ファイルロック取得のタイムアウト: 5秒
  - タイムアウト時のエラーハンドリング
  - リトライ機能（最大3回）
  - デッドロックの防止

- **NFR-004**: HTTPリクエストには適切なタイムアウトを設定すること
  - 接続タイムアウト: 10秒
  - 読み取りタイムアウト: 30秒
  - タイムアウト発生時のエラーログ出力
  - タイムアウト後の自動リトライ（オプション）

### 1.3 リソース効率

- **NFR-005**: メモリ使用量を最小限に抑えること
  - 不要なオブジェクトの適切な解放
  - 大きなデータのストリーミング処理
  - メモリリークの防止
  - スマートポインタの活用（Qt 5/6）

- **NFR-006**: CPU使用率を適切に管理すること
  - アイドル時のCPU使用率: 1%以下
  - 取得・投稿処理時のCPU使用率: 10%以下（通常状態）
  - 効率的なアルゴリズムの使用
  - 不要な計算の回避

### 1.4 応答時間

- **NFR-007**: ニュース記事取得から投稿完了までの時間
  - 1記事あたりの処理時間: 5秒以内（通常時）
  - ネットワーク遅延を除く内部処理時間: 1秒以内
  - タイムアウトを含む最大処理時間: 45秒以内

- **NFR-008**: システム起動時間
  - プロセス起動から動作開始まで: 3秒以内
  - 設定ファイル読み込み時間: 1秒以内
  - 初期化処理の最適化

## 2. 信頼性要件

### 2.1 エラー耐性

- **NFR-009**: ネットワークエラーが発生した場合でも、システムは継続動作すること
  - エラー発生時の適切なログ出力
  - エラーをスキップして次の処理へ移行
  - 致命的エラーと非致命的エラーの区別
  - 自動復旧機能

- **NFR-010**: 設定ファイルの破損を検出し、適切なエラーメッセージを出力すること
  - JSON構文エラーの検出
  - 必須フィールド欠損の検出
  - デフォルト値へのフォールバック
  - 管理者への通知（ログ出力）

- **NFR-011**: 複数のプロセスから同時に設定ファイルにアクセスしても、データの整合性が保たれること
  - ロックファイルによる排他制御
  - ACID特性の保証（Atomicity, Consistency, Isolation, Durability）
  - トランザクション処理の実装
  - ロック失敗時の適切なエラーハンドリング

- **NFR-012**: Webサイトの構造変更によりXPath抽出が失敗した場合でも、システムは継続動作すること
  - XPath抽出失敗の検出
  - エラーログへの詳細情報出力
  - 他のニュースソースからの取得継続
  - 管理者への通知（ログ出力）

### 2.2 可用性

- **NFR-013**: システムの稼働率
  - 目標稼働率: 99%以上
  - 計画的なメンテナンス時間を除く
  - 自動復旧機能による可用性向上

- **NFR-014**: 自動復旧機能
  - プロセスクラッシュ時のSystemdによる自動再起動
  - Restart=on-failure設定
  - RestartSec=30s設定
  - 最大再起動回数: 5回/10分

### 2.3 データ整合性

- **NFR-015**: ログファイルのデータ整合性を保証すること
  - 書き込み時の排他制御
  - アトミックな書き込み操作
  - 一時ファイルを使用した安全な更新
  - 書き込み失敗時のロールバック

- **NFR-016**: 設定ファイルの整合性を保証すること
  - 更新前のバックアップ作成
  - 検証済みデータのみ書き込み
  - 破損検出時の復旧機能
  - バージョン管理（オプション）

## 3. 保守性要件

### 3.1 設定の外部化

- **NFR-017**: XPath式、URL、取得間隔などの設定は全て外部設定ファイル（JSON）で管理されること
  - ハードコーディングの禁止
  - 設定項目の一元管理
  - 設定ファイルのドキュメント化
  - 設定例の提供

- **NFR-018**: ソースコードの変更なしに、新しいニュースソースを追加できる設計であること
  - プラグイン機構またはテンプレート方式
  - 設定ファイルへの追加のみで対応可能
  - スキーマの拡張性
  - 後方互換性の維持

### 3.2 ログとモニタリング

- **NFR-019**: ログファイルはJSON形式で、機械的に処理可能な形式であること
  - 構造化されたログデータ
  - パース可能なフォーマット
  - ログ解析ツールとの連携を想定
  - タイムスタンプの標準化（ISO 8601）

- **NFR-020**: エラー発生時には、原因を特定できる詳細な情報をログに出力すること
  - スタックトレース情報
  - エラーコード
  - 関連するパラメータ値
  - エラー発生時の状態情報
  - ネットワークエラー時のHTTPステータスコード

- **NFR-021**: ログレベルの設定
  - DEBUG: デバッグ情報の詳細出力
  - INFO: 通常の動作情報
  - WARNING: 警告レベルのイベント
  - ERROR: エラー情報
  - CRITICAL: 致命的なエラー
  - 設定ファイルでのログレベル指定

### 3.3 コードの保守性

- **NFR-022**: ソースコードはC++17標準に準拠すること
  - モダンC++の機能活用
  - auto キーワードの使用
  - ラムダ式の活用
  - スマートポインタの使用

- **NFR-023**: コードコメントとドキュメント
  - 主要な関数・クラスにコメント記載
  - Doxygen形式のドキュメントコメント
  - README.mdの充実
  - インラインコメントの適切な使用

- **NFR-024**: コーディング規約の遵守
  - 一貫したインデント（スペース4つ）
  - 命名規則の統一
  - クラス名: PascalCase
  - 関数名: camelCase
  - 変数名: camelCase
  - 定数名: UPPER_SNAKE_CASE

## 4. 移植性要件

### 4.1 対応OS

- **NFR-025**: Linux環境で動作すること
  - Red Hat Enterprise Linux 7以降
  - SUSE Linux Enterprise 12以降
  - openSUSE Leap 15以降
  - Debian 11以降
  - Ubuntu 18.04 LTS以降
  - Raspberry Pi OS（Raspbian）

- **NFR-026**: 32ビット/64ビットアーキテクチャ対応
  - x86（32ビット）
  - x86_64（64ビット）
  - ARM（Raspberry Pi）
  - ARM64

### 4.2 Qtバージョン対応

- **NFR-027**: Qt 5.15以降、またはQt 6.x系で動作すること
  - Qt 5.15 LTS対応
  - Qt 6.2 LTS以降対応
  - バージョンによる条件分岐の最小化
  - 非推奨APIの使用回避

- **NFR-028**: Qt 5ライブラリを同梱することで、Qtがインストールされていない環境でも動作可能であること
  - 静的リンクまたは動的ライブラリの同梱
  - LD_LIBRARY_PATH設定
  - ラッパースクリプトによる環境設定
  - 依存ライブラリの明示

### 4.3 ビルドシステム

- **NFR-029**: CMakeによるビルドシステムを採用し、異なる環境でも容易にビルドできること
  - CMake 3.16以上対応
  - クロスプラットフォームビルド
  - 外部依存の自動検出
  - ビルドオプションの提供

- **NFR-030**: ビルド時の依存関係を明確にすること
  - CMakeLists.txtへの依存関係記載
  - find_package()による自動検出
  - PKG-CONFIGファイルの提供
  - ビルドドキュメントの整備

### 4.4 依存ライブラリ

- **NFR-031**: 必須ライブラリの最小化
  - Qt（Core, Network）
  - libxml2
  - OpenSSL 3（Qt 6使用時）
  - 標準Cライブラリ

- **NFR-032**: ライブラリバージョンの互換性
  - 古いバージョンへの対応
  - バージョン検出機能
  - 条件付きコンパイル
  - 後方互換性の維持

## 5. セキュリティ要件

### 5.1 認証情報の管理

- **NFR-033**: News APIキー等の認証情報は設定ファイルで管理し、ソースコードに埋め込まないこと
  - 環境変数での指定も可能
  - 設定ファイルのパーミッション管理（600推奨）
  - 認証情報の暗号化（オプション）
  - バージョン管理システムへのコミット防止

- **NFR-034**: 設定ファイルの適切な権限設定
  - 推奨パーミッション: 600（所有者のみ読み書き可能）
  - 所有者: root または実行ユーザー
  - グループ: root または適切なグループ
  - その他: アクセス不可

### 5.2 通信セキュリティ

- **NFR-035**: HTTP通信においてHTTPSをサポートし、OpenSSL 3を使用すること
  - TLS 1.2以降の使用
  - 証明書の検証
  - 中間者攻撃の防止
  - 適切な暗号スイートの選択

- **NFR-036**: SSL/TLS証明書の検証
  - 証明書チェーンの検証
  - 有効期限の確認
  - ホスト名の検証
  - 自己署名証明書の扱い（設定可能）

### 5.3 入力検証

- **NFR-037**: 外部入力（ニュース記事の内容）を掲示板に投稿する際、適切なエンコーディング処理を行うこと
  - XSS攻撃の防止
  - SQLインジェクションの防止（該当する場合）
  - コマンドインジェクションの防止
  - 特殊文字のエスケープ処理

- **NFR-038**: 設定ファイルの入力検証
  - JSON構文の検証
  - データ型の検証
  - 値の範囲チェック
  - 不正な値の拒否

### 5.4 権限管理

- **NFR-039**: 最小権限の原則
  - root権限での実行を避ける
  - 専用ユーザーでの実行推奨
  - 必要最小限のファイルアクセス権限
  - Systemdのユーザー指定

- **NFR-040**: ファイルシステムへのアクセス制限
  - 設定ファイルディレクトリ: /etc/qNewsFlash/
  - ログファイルディレクトリ: /var/log/
  - 実行ファイル: /usr/local/bin/ または /usr/bin/
  - 一時ファイル: /tmp/ または /var/tmp/

### 6.3 ドキュメント
