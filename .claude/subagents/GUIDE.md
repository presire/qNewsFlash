# Subagents Guide

サブエージェントは、特定の専門領域に特化した自律的なAIアシスタントです。
独立したコンテキストウィンドウを持ち、複雑なタスクを委譲できます。

---

## 利用可能なサブエージェント

| サブエージェント | 専門領域 | 使用場面 |
|------------------|----------|----------|
| qt-cpp-developer | Qt5/Qt6 C++開発 | Qt Network、Signal/Slot、メモリ管理 |
| xpath-specialist | XPath/Webスクレイピング | ニュースサイト構造変更、XPath式作成 |
| cmake-builder | CMake/ビルド | 依存関係解決、クロスコンパイル |
| research-collector | 技術調査・資料収集 | Qt/libxml2 API調査、公式ドキュメント分析 |

---

## 使用方法

### タスク委譲

サブエージェントに複雑なタスクを委譲できます：

```
qt-cpp-developerサブエージェントに、HTTP通信クラスの実装を委譲してください。
```

```
xpath-specialistサブエージェントに、時事ドットコムの新しいXPath式の作成を依頼してください。
```

```
cmake-builderサブエージェントに、Qt6用のビルド設定を修正させてください。
```

### 自動選択

タスク内容に基づいて適切なサブエージェントが自動選択されます：

| トリガー語/タスク | 選択されるサブエージェント |
|-------------------|----------------------------|
| Qt Network API実装、Signal/Slot | qt-cpp-developer |
| エンコーディング変換、タイマー実装 | qt-cpp-developer |
| XPath式の作成・修正、HTML構造分析 | xpath-specialist |
| ニュースサイト追加、スクレイピング | xpath-specialist |
| CMakeビルドエラー、find_package | cmake-builder |
| クロスコンパイル、インストール設定 | cmake-builder |
| API調査、公式ドキュメント | research-collector |
| 技術比較、ベストプラクティス調査 | research-collector |

## サブエージェントとスキルの違い

| 特徴 | スキル | サブエージェント |
|------|--------|------------------|
| コンテキスト | 共有 | 独立 |
| 実行形態 | ワークフロー | 自律的エージェント |
| 呼び出し | 明示的/自動 | タスク委譲 |
| 用途 | 手順・知識の提供 | 複雑なタスクの委譲 |
| 応答速度 | 即座 | 処理時間あり |
| 対話 | 不可 | 可能 |
| 継続性 | なし | セッション継続可能 |

**使い分け:**
- パターンやベストプラクティスを知りたい → スキル
- 複雑な実装や調査を任せたい → サブエージェント

## 各サブエージェントの詳細

### qt-cpp-developer

Qt5/Qt6 C++開発を専門とするエージェント。

**得意なタスク:**
- QNetworkAccessManager (HTTP/HTTPS通信)
- QString / QTextCodec (文字列処理、エンコーディング)
- QTimer (定期実行)
- Smart Pointers (QSharedPointer, QScopedPointer)
- Signal/Slot パターン
- Qt5/Qt6 互換性対応
- ネットワークリトライ処理
- メモリリーク防止

**タスク受領時の確認項目:**
- 目的の明確化
- Qt バージョン要件
- ネットワーク要件
- エンコーディング要件

**詳細:** `qt-cpp-developer.md`

### xpath-specialist

XPath クエリと Web スクレイピングを専門とするエージェント。

**得意なタスク:**
- XPath 1.0 式の作成・最適化
- libxml2 API の使用
- HTML/XML パース処理
- ニュースサイト構造分析
- Webサイト構造変更対応
- エンコーディング問題解決

**分析フロー:**
1. HTML 取得
2. 構造分析
3. XPath 作成
4. 検証

**詳細:** `xpath-specialist.md`

### cmake-builder

CMake ビルドシステムを専門とするエージェント。

**得意なタスク:**
- CMakeLists.txt の作成・修正
- find_package (Qt, libxml2, OpenSSL)
- ビルドオプション設定
- クロスコンパイル設定
- インストール設定
- CI/CD 連携

**診断フロー:**
1. エラー分類
2. 原因特定
3. 解決策提示

**詳細:** `cmake-builder.md`

### research-collector

技術調査・資料収集を専門とするエージェント。

**得意なタスク:**
- Qt API リファレンス調査
- libxml2 ドキュメント分析
- XPath ベストプラクティス収集
- 技術比較調査
- エラー原因調査

**詳細:** `research-collector.md`

## ワークフロー例

### 1. 新しいニュースソース追加フロー

```
1. research-collector
   - 対象サイトの構造調査
   - RSS フィードの有無確認
   - エンコーディング確認

2. xpath-specialist
   - HTML 構造分析
   - XPath 式の作成
   - テストと検証

3. qt-cpp-developer
   - 新しいクラスの実装
   - HTTP 通信の実装
   - エラーハンドリング

4. cmake-builder
   - CMakeLists.txt の更新
   - ビルド確認
```

### 2. ビルド問題解決フロー

```
1. cmake-builder
   - エラーメッセージの分析
   - 原因の特定 (パッケージ不足/パス問題/設定ミス)
   - 解決策の提示

2. research-collector (必要に応じて)
   - 不足パッケージのインストール方法調査
   - 公式ドキュメントの確認

3. cmake-builder
   - 設定の修正
   - ビルドの再実行
   - 成功確認
```

### 3. Webサイト構造変更対応フロー

```
1. xpath-specialist
   - 変更の検知と分析
   - 新しい HTML 構造の確認
   - 新しい XPath 式の作成

2. xpath-specialist
   - テスト HTML での検証
   - 設定ファイルの更新
   - 動作確認

3. linux-systemd (スキル)
   - サービスの再起動
   - ログでの動作確認
```

### 4. Qt6 移行フロー

```
1. research-collector
   - Qt5 → Qt6 移行ガイド調査
   - API 変更点の確認
   - 互換性情報の収集

2. qt-cpp-developer
   - コードの互換性確認
   - 条件コンパイルの実装
   - 非推奨 API の置換

3. cmake-builder
   - Qt6 用ビルド設定
   - OpenSSL 3 依存関係の解決
   - ビルドとテスト
```

### 5. クロスコンパイル環境構築フロー

```
1. research-collector
   - ターゲットプラットフォームの調査
   - 必要なツールチェーンの確認

2. cmake-builder
   - ツールチェーンファイルの作成
   - クロスコンパイル設定
   - ビルドとテスト

3. qt-cpp-developer (必要に応じて)
   - プラットフォーム固有のコード修正
   - パフォーマンス最適化
```

## エスカレーション基準

### Oracle への相談タイミング

以下の場合は Oracle エージェントへの相談を検討してください：

| 状況 | 理由 |
|------|------|
| アーキテクチャ決定 | 複数システムに影響する設計判断 |
| 2回以上の修正失敗 | 根本的な問題の見落とし可能性 |
| セキュリティ懸念 | 認証、暗号化、データ保護に関わる実装 |
| パフォーマンス問題 | ボトルネックの特定と解決 |
| 複雑な依存関係 | 複数モジュール間の相互作用 |

### 相談の流れ

```
1. サブエージェントで問題を試行
   ↓
2. 解決しない場合、問題を整理
   ↓
3. Oracle に相談
   - 問題の概要
   - 試したこと
   - エラーログ/コード
   ↓
4. Oracle のアドバイスに基づいて対応
```

### 相談すべきでないケース

- 単純な構文エラー
- ドキュメントで明確な API 使用法
- 既知のパターンに従う実装
- 単一ファイルの小さな変更

## サブエージェント連携のベストプラクティス

### 1. タスクの明確な定義

```
良い例:
「xpath-specialistに、時事ドットコムの速報ページから
タイトル、URL、日時を抽出するXPath式を作成させてください。
エンコーディングはShift-JISです。」

悪い例:
「XPathを作って」
```

### 2. コンテキストの提供

```
良い例:
「qt-cpp-developerに、既存のHtmlFetcherクラスに
タイムアウト機能を追加してください。
現在の実装は300行で、QNetworkAccessManagerを使用しています。」

悪い例:
「タイムアウトを追加して」
```

### 3. 期待する出力の指定

```
良い例:
「cmake-builderに、Qt6用のビルドエラーを解決させてください。
期待する出力は、修正後のCMakeLists.txtと、
必要なパッケージのインストールコマンドです。」

悪い例:
「ビルドエラーを直して」
```

### 4. セッションの継続利用

サブエージェントはセッションを継続できます。
同じエージェントに追加のタスクを依頼する場合は、
セッションIDを維持することで、コンテキストを保持できます。

## トラブルシューティング

### サブエージェントが適切に選択されない

**原因:** タスクの記述が曖昧
**解決:** 専門用語を明確に使用

```
× 「処理を実装して」
○ 「QNetworkAccessManagerでHTTP GETを実装して」
```

### サブエージェントの出力が不十分

**原因:** コンテキスト不足
**解決:** 関連ファイル、エラーメッセージ、期待する出力を明示

### 複数のサブエージェントが必要な場合

**解決:** ワークフローに従って段階的に委譲
